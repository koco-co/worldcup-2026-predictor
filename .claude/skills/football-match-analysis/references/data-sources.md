# 联网取数手册（数据源分层 · 搜索词 · 不可得时如何降级）

SKILL.md 要求**联网取数是强制前置步骤**（不是「能联网就用」的可选项）。本文件给出**优先去哪取、次选/备选是什么、拿不到怎么办**——目的就是让你**直奔优先源、少做大量盲目 WebFetch**。**每次正式分析前先读本文件，按下面「分层源表」直奔优先源。**

## 取数原则

1. **多源交叉**：关键信息（伤停、首发、盘口）至少看两家，不一致时取主流口径并**标注分歧**，不要只信一家。
2. **诚实优先于完整 + 真实数据必须带可信外链**：查得到的用真实数据，并用 **Markdown 外链 `[数据/出处](URL)`** 标注可信网页或官方接口来源、尽量注明取数时间；查不到的**明确标「数据不可得 / 为假设值」**，基于合理假设分析并说明，**绝不把假设写成确定数据，也不给无源的具体数字**。
   - 天气：`比赛日 28°C、多云（[AccuWeather](https://...)）`
   - 赔率：`胜平负 1.34/4.70/5.90（[体彩官方](https://m.sporttery.cn/...)，截至 17:41）`
   - 伤停/近况：`核心后腰停赛（[ESPN](https://...)）`
3. **下面给的是分层网址清单（优先/次选/备选），但站点会改版/改名、盘口与可串场次随时变**：优先源失效或被挡，就顺位换次选 / 备选，别死磕单一源；以官方与实时页面为准。

## 分层源表（优先 → 次选 → 备选 · 直奔优先源，少做盲目 WebFetch）

**用法：** 每类数据**先打优先源**，拿到就停；优先源失效 / 被挡 / 信息缺，再顺位次选、备选。带 ✅ 的是 2026-06 实测可达（一两次 `WebSearch`/`WebFetch` 即可拿到）。

**实战经验（来自真实试跑）：**
- **最易得**：时间地点、双方近况、历史交锋、欧赔 1X2、天气——一次普通搜索 + 多源印证即可。
- **中等**：伤停大方向能拿到，但**不同站点阵型/边锋人选常不一致**（如 4-2-3-1 vs 4-2-4），取主流并标分歧。
- **最难 / 常拿不到**：① 即时亚盘的**初盘→即时升降方向**（普通网页摘要只给一个静态点）；② 赛前**主裁**（常未公布）。**xG/xGA 不在此列**——WebSearch 摘要难拿，但进 Understat / `soccerdata` / FBref 可取（国家队/赛季初样本太小时意义有限）。搜「Asian handicap」时返回的多是博彩推广页，命中率低——优先直接进 Oddschecker / Understat 这类源站。

## 实测可达性（联网工具实测，2026-06）

用 WebSearch / WebFetch 实测过，结论：

- **天气、举办城市环境**：✅ 容易。AccuWeather 有 2026 世界杯专题页给逐城逐场预报；普通天气搜索也能拿到（如某场 24°C、0% 降雨、湿度 54%）。注意：赛前 ~14 天才有靠谱预报，更远只能给「气候常态」。
- **球员近况 / 伤停**：✅ 容易。ESPN 伤停 tracker、FIFA 官网、各国足协 / Bundesliga 给的是最新动态（伤缺、替补入替、复出首战等都能逐条拿到），多源交叉即可。
- **竞彩 / 体彩赔率（胜平负、让球、比分、总进球、半全场）**：✅ **能拿，用官方接口**。体彩官方页面 `m.sporttery.cn/mjc/jsq/zqspf/` 是 JS 壳抓不到，但其**背后的官方数据接口 `webapi.sporttery.cn` 实测带请求头后返回完整实时 JSON**：含五种玩法赔率、让球数、各玩法更新时间，**且每个赔率带升降标志位**。详见下节「竞彩官方数据源速查」。坑：① 缺请求头会 **403**（实测裸 WebFetch 被挡，带 `User-Agent`+`Referer` 用 MCP fetch_json 才通）；② **带正确请求头仍可能返回 HTTP 567**（实测 2026-06 多次出现，疑似服务端/地域限制或非美区出口被挡，与缺头的 403 不同，重试也回不来）——此时**最多重试 1–2 次即放弃**，回退欧赔换算并标「估计值」，**别反复重试浪费时间**；③ 接口非公开、可能限流/变动，失效时回退抓中彩网/500 页面。
- **竞彩官方让球的升降方向**：✅ 能拿。官方接口里每个赔率字段都跟一个 `f` 标志（`1` 升 / `-1` 降 / `0` 平），即 SP 红绿方向。**这是竞彩官方源相对外围亚盘的优势**。
- **xG / xGA**：WebSearch 摘要仍难拿，但 **Understat 公开数据可取**（`soccerdata` 库 / Understat `get_event_xg`，带射门数据），能进站或调库时优先用，作为攻防对比与「xG 回归均值」判断的底座；仍拿不到时按下面话术标注假设值。
- **外围亚盘细盘水位 / 主裁**：❌ 仍难。外围亚盘升降抓到的多是快照；主裁赛前常未公布。这两项按下面话术标注。

**取数动作建议：** 易得项（天气、伤停、近况、欧赔）用 `WebSearch`；**竞彩赔率用官方接口**（`mcp__fetch__fetch_json` 带请求头，见下节）；xG 进 Understat / `soccerdata` 可取；真正拿不到的（外围升降水、主裁）老实标注，不要编。

## 竞彩官方数据源速查（体彩 sporttery）

用户指定的竞彩源：
- 胜平负计算器页：`https://m.sporttery.cn/mjc/jsq/zqspf/`
- **赛程页（看尚未开始的比赛）：`https://m.sporttery.cn/mjc/zqsj/?tab=concern`**

两个都是 JS 壳，**不要直接抓页面**，要抓它们背后的官方接口（下面的 `getMatchCalculatorV1` 接口本身就返回 `matchStatus":"Selling"` 的未开赛比赛，已覆盖「赛程」需求）。竞彩每场在售的就是**五大玩法**：胜平负、让球胜平负、比分、总进球数、半全场胜平负（**无大小球**）。

**接口（实测可用，2026-06）：**
```
https://webapi.sporttery.cn/gateway/jc/football/getMatchCalculatorV1.qry?poolCode=had,hhad,crs,ttg,hafu&channel=c
```
- `poolCode` 按需取，逗号分隔：`had`=胜平负、`hhad`=让球胜平负、`crs`=比分、`ttg`=总进球、`hafu`=半全场。只要胜平负+让球用 `had,hhad` 即可。
- **必须带请求头，否则 403：** `User-Agent`（移动端 UA）+ `Referer: https://m.sporttery.cn/mjc/jsq/zqspf/`。用 `mcp__fetch__fetch_json` 传 `headers` 最稳；`WebFetch` 不能自定义头，易被挡。
- **带头仍可能 HTTP 567（重要，别死磕）：** 实测 2026-06 即便带齐 UA+Referer，接口仍可能连续返回 **HTTP 567**（疑似服务端/地域限制，与缺头的 403 不是一回事，多次重试也回不来）。遇到 567 **最多重试 1–2 次就放弃**，直接回退到欧赔换算（Oddschecker/OddsPortal/新浪/500）并把赔率标为「估计值、以官方开奖 SP 为准」——**不要重试 4 次以上空耗**。

**返回字段映射**（`value.matchInfoList[].subMatchList[]` 每场一项）：
- 对阵/赛事：`homeTeamAllName` / `awayTeamAllName` / `leagueAllName`（如「世界杯」）/ `matchNumStr`（如「周六033」）/ `matchDate` / `matchTime` / `matchStatus`。
- `had`：`h`/`d`/`a` = 主胜/平/客胜赔率；`hf`/`df`/`af` = 升降标志（`1` 升 / `-1` 降 / `0` 平）；`updateTime` = 该盘更新时间。
- `hhad`：`h`/`d`/`a` + `goalLine`（让球数，如 `-1`）+ 同样的 `*f` 升降标志。
- `crs`：`sXXsYY` = 比分 X:Y 的赔率（如 `s02s01`=2:1）；`s1sh`/`s1sd`/`s1sa` = 胜其他/平其他/负其他。
- `ttg`：各总进球档赔率；`hafu`：`hh`/`hd`/`ha`/`dh`/`dd`/`da`/`ah`/`ad`/`aa` = 半全场 9 组合。
- **单关/过关标识（在 `subMatchList[].poolList[]` 每个玩法对象里，决定该玩法能否单买，务必看）：**
  - `single` / `bettingSingle`：`1` = 该玩法**可单关（可单场单买）**；`0` = **仅过关（必须跨场串关，不能单买）**。
  - `bettingAllUp`：`1` = 可参与过关/串关。
  - **实战经验**：热门/悬念场次的**胜平负（`had`）、让球（`hhad`）常被设为仅过关（`single:0`）**，而**比分（`crs`）、总进球（`ttg`）、半全场（`hafu`）多可单关（`single:1`）**；个别场次甚至不挂胜平负盘（`had` 为空 `{}`）。**推荐"主胜/让胜"这类方向前，先查该场该玩法的 `single` 标识**——若仅过关，就只能写成 2 串 1 等组合，不能当单关推荐（单关资格以官方实时为准、会变动）。

**诚实标注：** 接口为非公开调用、可能限流或改版；赔率为某时刻快照，最终以**官方开奖 SP 为准**。接口失效时回退到中彩网/500/okooo 抓页面（能拿胜平负/让球点值，但升降不一定有）。

## 不可得时的标注话术（直接套用）

- 整项缺失：`数据不可得，以下基于合理假设分析。`
- 盘口只有静态值：`仅即时静态值，初盘与升降方向不可得，无法判断升降盘倾向。`
- 样本过小（国家队 / 赛季初 / 杯赛）：`样本过小（仅 N 场），场均类指标参考意义有限。`
- 多源分歧：`各源阵型/人选不一致，此处取主流 <X>，另有 <Y> 说法。`

## 盘口表「双列」的现实处理

模块四的「亚盘初盘/即时盘」「欧赔变化」「大小球初盘/即时盘」三张表默认填「初盘 vs 即时盘」两列。但散户能联网拿到的**几乎只有即时一个静态值**，初盘与升降常永远不可得。此时**不要硬留半张空表**：

> 拿不到初盘/即时对比时，把两列合并成单列「**当前盘口 + 解读**」，并注明「升降方向不可得」即可——保持信息密度，避免满屏「数据不可得」。

## 降级模式（仅当工具确实不可用：断网 / 接口持续报错且换源仍失败）

联网是**强制前置步骤**——**只有工具确实用不了才进降级模式，「懒得查」不算理由**。进入降级模式时，严格执行 SKILL.md「降级模式护栏」：

1. 报告**最顶部**打「⚠️ 降级模式 · 未联网 / 数据滞后」，所有具体数字标「估计值」、缺口标清楚。
2. **不得凭假设判对手 A/B/C**——无真实近况就标「类型待核实」、按中性处理，**不默认判 B 纪律铁桶**。
3. **不得触发高相关批次收缩、不得据假设把保底层缩成单场或砍总投入**（GIGO 降级模式示例：三场无数据默认判 B → 误触发收缩 → 保底缩单场 + 砍仓，属误触发，禁止）。
4. 直接用用户提供的数据，基于合理假设分析并声明数据可能滞后；不要假装拥有不存在的实时盘口或 xG。
